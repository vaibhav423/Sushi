# ChrootBridge Limitations

Known constraints, edge cases, and why they exist.

## 1. No ContentResolver APIs (`app_process` vs AMS)
`ContentResolver` (used by `Settings.Secure`, `WifiManager.getConnectionInfo`, `ContactsContract`) requires the calling process to be registered with `ActivityManagerService` as a live Android app via `ActivityThread`. Since `app_process` daemons bypass the Android app lifecycle, AMS rejects the IPC with *"Unable to find app for caller"*.

- **Workaround**: delegate to `asu -c 'settings get ...'`, or wait for the LSPosed SystemUI module (which runs inside a real app process).

## 2. Multi-Class Definitions Crash (ART `ClassLoader.defineClass`)
BeanShell's `BSHClassDeclaration` tries to compile nested class definitions into JVM bytecode and load them via `ClassLoader.defineClass()`. Android's ART throws `UnsupportedOperationException: can't load this type of class file` for non-dex bytecode generated by BeanShell's internal compiler.

- **Impact**: You cannot define classes inside `.java` files run through `droid -f`. Use only top-level statements, methods, and expressions.
- **Workaround**: Use interfaces, anonymous classes, or pre-compiled DEX files loaded via a custom classloader.

## 3. Persistent BeanShell Namespace Pollution
The daemon uses a single persistent `Interpreter` instance. All `java` and `java-raw` commands share the same namespace. If command A declares `String s = "hello"` and command B declares `Set<String> s = ...`, the second fails with *"variable s was previously declared with type: java.lang.String"*.

- **Workaround**: Use unique variable names per command, or restart the daemon (`bridge restart`) to clear the interpreter state.
- **Future fix**: Spawn a fresh `Interpreter` per command (trade-off: loses state across commands).

## 4. Limited Character Set for Input Injection
`injectText()` only handles: `a-z`, `0-9`, `enter`, `space`, `.`, `,`, `/`. Characters outside this set (symbols, uppercase with shift, accented chars, emoji) are silently skipped.

- **Workaround**: For complex text, use `droid java-raw` with BeanShell to inject via platform APIs directly, or use Android's `adb shell input text` as a fallback.

## 5. Socket Protocol is Newline-Delimited
The daemon reads commands with `BufferedReader.readLine()`, which truncates at the first `\n`. Any command containing a literal newline (e.g., multi-line clipboard text) will be cut off.

- **Fix applied**: The `clipboard-set` command accepts Base64-encoded payloads to bypass the delimiter. `droid clipboard set "text"` automatically Base64-encodes on the client side.
- **Impact**: All commands are inherently single-line. Multi-line data must be encoded.

## 6. UID 2000 is Required (Not Root, Not System)
- UID 0 (root): Killed by Android kernel security layer when running `app_process`.
- UID 1000 (system): Lacks clipboard/input background exemptions that UID 2000 (shell) natively gets.
- UID 2000: The only UID that can read/write clipboard and inject input from a background `app_process` daemon, because Android hardcodes these exemptions for the ADB shell user.

- **Implication**: The `runas2000` setuid binary is mandatory. Dropping to UID 2000 must happen before `exec`-ing `app_process`.

## 7. ART Dies Without `BOOTCLASSPATH`
The ART runtime crashes with `SIGABRT` during initialization if `BOOTCLASSPATH` or `DEX2OATBOOTCLASSPATH` environment variables are missing or empty. These must be scraped from the running `zygote64` process's `/proc/<pid>/environ` before starting the daemon.

- **Handled by**: `bridge.sh` reads these from zygote64 and exports them before launching.

## 8. Daemon Can't Survive Without `exec` + `disown`
Android's `su` process management kills backgrounded child processes when the `su` session exits. Standard `nohup` or `&` is insufficient.

- **Required pattern**: `asu -c 'exec app_process ...' &; disown`
- The `exec` replaces the `su` shell itself with the daemon, and the outer `&` + `disown` detaches the parent shell.

## 9. No System-Wide Dialogs from `app_process`
Attempting to show `WindowManager.LayoutParams.TYPE_SYSTEM_ERROR` or similar overlay windows from `app_process` crashes `system_server` (the process hosting WindowManager). `app_process` lacks the window token and permission context that a real Android app window requires.

- **Delegated to**: Future LSPosed SystemUI module, which can draw system dialogs because it runs inside an actual Android process with proper window tokens.

## 10. droid Client Response Buffer is Fixed (4096 bytes)
The C client's `send_and_recv()` reads responses in 4096-byte chunks. While it loops until EOF (so large responses aren't truncated), the fixed buffer size is a potential fragility if the daemon doesn't close the connection cleanly.

## 11. No User Script Isolation
All BeanShell scripts (`droid java`, `droid -f`) run in the same JVM process as the daemon itself. A malicious or buggy script can:
- Crash the daemon (caught by the broad `try/catch` in `handleCommand`, but infinite loops will block the thread)
- Access/modify internal daemon state via reflection on `sContext`, `sBshInterpreter`, etc.
- Exhaust resources (memory, threads)

## 12. Android-Only (No Desktop JVM Compatibility)
The daemon uses Android-specific APIs (`android.app.ActivityThread`, `LocalServerSocket`, `InputManager`) and requires an ART runtime. It cannot run on standard desktop JVMs for testing.

## 13. `build/` Output Directory Required
The Makefile and deploy scripts expect compiled artifacts in `build/`. Running `make` from outside the project root or without the `build/` directory existing will fail (the Makefile now auto-creates it via an order-only prerequisite).
